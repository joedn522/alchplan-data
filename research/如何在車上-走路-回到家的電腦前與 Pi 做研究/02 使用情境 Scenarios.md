# 02 使用情境（Scenarios）

> 本頁專注：把「散步／開車／回到電腦前」變成具體場景，之後技術設計都要服務這些場景。

## 情境 A：出去散步
- 有一大段時間可以思考、聯想、把腦中東西倒出來。
- 手不一定在鍵盤上（可能只拿手機）。
- 又或者我可以把心得告訴你，然後你用語音回我，或是打電話給我。@pi

### A.1 故事版 user story
- 晚上散步 20 分鐘，我戴著耳機、拿著手機：
  1. 打開常用的錄音／語音備忘錄 App，按一下開始錄音。
  2. 對 Pi 說：「幫我起一個研究：今天想到 XXX、我在想 YYY。你先幫我列出幾個研究問題就好。」
  3. 錄完直接關掉畫面，繼續走路，不用管後處理。
  4. 回到家，Pi 已經把這段語音轉成文字，放進 Obsidian 的 `Voice-Inbox/` 並且在對應研究 note 底下開一個「今天散步想法」小節。
  5. 我打開電腦，只需要看整理好的問題清單，決定要不要繼續展開。

### A.2 風險 / 成本 / 成功畫面
- 風險：
  - 語音辨識錯字導致關鍵名詞錯掉。
  - 散步時網路不穩，無法即時上傳，但這其實可以接受（離線錄、回家再傳）。
- 成本：
  - 語音轉文字的運算成本（可由本機 Whisper 分攤，token 成本主要在後續 LLM）。
  - 我自己要稍微檢查重點有沒有被誤聽。
- 成功畫面：
  - 散步結束後，研究 note 裡自動多了一個「散步想法」區塊，裡面是整理好的 bullet 而不是一整坨未整理文字。

## 情境 B：開車
- 幾乎沒辦法看螢幕、打字，只能用語音（或完全離線）。

### B.1 故事版 user story
- 白天開車 40 分鐘通勤：
  1. 上車前或停紅燈時，按一下方向盤上的語音助理鍵，或直接叫出手機語音助理。
  2. 說：「幫我開一個 Pi 研究任務：題目是 XXX，目的是 YYY。」
  3. 系統把這段語音只當成『任務描述』記錄下來，不強求 Pi 立即回覆。
  4. 抵達目的地後，語音檔被自動上傳到電腦上的「研究收集管道」；
  5. 稍晚 Pi 把這段通勤語音整理成幾個具體任務（例如：定義研究問題、列出需查找的文獻），寫進同一篇研究 note。

### B.2 風險 / 成本 / 成功畫面
- 風險：
  - 最大風險其實是安全：不能為了想講清楚而分心。
  - 網路中斷 → 先本機存檔即可。
- 成本：
  - 若要即時對話（像打電話），就會需要穩定網路 + 持續的 ASR/LLM/TTS 運作，token 成本會偏高。
- 成功畫面：
  - 開車過程就像在「口頭寫研究日誌」，但我不需要當場看任何螢幕；研究進度晚上回家就已整理好。

## 情境 C：回到家的電腦前
- 可以看螢幕、打字、管理 Obsidian。

### C.1 故事版 user story
- 晚上坐回桌前：
  1. 打開 Obsidian，切到這個研究的主畫面。
  2. 可以看到今天散步、開車、白天想到的語音都已經轉成文字，整理在對應的 section 裡。
  3. 我挑一兩個最想繼續的線索，在 note 裡加上 `@pi` 標記，描述：「請幫我把這段變成一個小節，並查 3 篇參考資料。」
  4. Pi 根據這些 `@pi` 任務，去查資料、寫草稿，直接更新這一篇研究 note。
  5. 如果我想用「聽的」回顧，Pi 可以把目前草稿的摘要丟給一個 TTS / 語音 agent（例如 NotebookLM / Gemini），讓我邊做別的事情邊聽。

### C.2 延伸：NotebookLM / Gemini 語音對談
- 我完成一輪書寫後：
  - 可以把這篇研究 note 或其摘要丟進 NotebookLM，讓它生成更長篇的「語音講解版」。
  - 或由 Pi 先整理出「我應該跟 Gemini 確認的 3–5 個重點問題」，塞進一個 Gemini 語音 Session，
    - 我用語音和 Gemini 互動，釐清觀念與盲點；
    - 對話完後讓 Gemini 給一份 summary，我再貼回 Obsidian，Pi 幫我統整到研究主檔裡。

### C.3 風險 / 成本 / 成功畫面
- 風險：
  - 資訊分散在 Pi、Gemini、NotebookLM 三個地方，需要一個「最後都回到 Obsidian」的習慣。
- 成本：
  - NotebookLM / Gemini 的使用成本 + LLM 整理成本；
  - 但這些多半發生在「深度整理階段」，不是每天 24 小時一直跑。
- 成功畫面：
  - 研究主檔永遠是 Obsidian 那一篇；
  - 其他語音/LLM 只是不同的「閱讀與理解介面」，最終成果都收斂回來。
