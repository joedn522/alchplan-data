# 03 技術構面（Technical Dimensions）

> 本頁專注：盤點輸入管道、處理模式、以及最後怎麼落在 Obsidian 裡。

## 3.0 什麼是 ASR？（Automatic Speech Recognition）

ASR（Automatic Speech Recognition，自動語音辨識）就是「把人講的話，轉成文字」的系統。你可以把它想成是一個專門聽聲音、輸出文字的模型或服務。

- **輸入：** 麥克風收到的語音（你講話的聲音，通常是幾秒到幾分鐘的音訊）。
- **輸出：** 對應的文字稿（有時會帶時間戳記，知道哪個字是幾秒時說的）。

實際使用時，ASR 通常有幾種工作模式：

1. **離線批次轉錄**  
   - 做法：先錄好一整段音檔，再丟給 ASR 一次轉完。  
   - 例子：你走完路回家，把 10 分鐘語音檔丟給 Whisper，幾十秒～一兩分鐘內轉成完整文字。  
   - 適合：會議錄音、長篇日記、Podcast 轉稿。

2. **即時／串流轉錄（Streaming）**  
   - 做法：一邊說話，一邊把音訊切成小塊丟給 ASR，讓它持續更新文字。  
   - 例子：語音助理、線上會議字幕、你理想中的「打電話給 Pi」。  
   - 需要：
     - 足夠快的運算（GPU 像 5080 很適合這種工作）。
     - 設計好分段與緩衝（避免一半句子就被送出）。

3. **雲端 ASR 服務**  
   - 做法：把音訊上傳到雲端 API，換回文字。  
   - 優點：設定簡單、效果通常不錯。  
   - 缺點：
     - 需要穩定網路；
     - 會有隱私與成本（每分鐘／每 token 收費）考量。

在你的研究裡，ASR 的角色就是：

> 「把散步／開車時的語音（或電話裡的聲音）變成 Pi 能理解、能塞進 Obsidian 的文字。」

技術上可以選擇：
- 本機 ASR（例如：Whisper 在 Mac mini 或 Windows + 5080 上跑）；
- 雲端 ASR（例如各家語音轉文字 API）。

你之後要決定的是：
- 哪些情境一定要即時（需要 Streaming ASR）；
- 哪些情境只要晚上批次轉錄就好（可以用離線批次 ASR）。

## 3.1 輸入管道（我 → Pi）

### 語音 → 文字

1. **鍵盤語音輸入 → 聊天（Telegram 等）**
   - 流程：
     - 手機鍵盤麥克風 → 語音轉文字 → 文字訊息直接送給 Pi。
   - 優點：
     - 不用額外 App，延遲低，適合短句、指令型請求。
   - 缺點：
     - 長篇講話時，容易斷句／誤字，需要我回頭修正。

2. **語音備忘錄 / Just Press Record → iCloud → Whisper → Obsidian `Voice-Inbox/`**
   - 流程：
     1. 手機上按錄音，隨意講 1–5 分鐘想法。
     2. 錄音自動同步到 iCloud 資料夾。
     3. Mac mini 上的監控程式（Whisper pipeline）偵測新檔 → 轉文字。
     4. 把轉好的文字存成 Obsidian 的 `Voice-Inbox/*.md`。
   - 優點：
     - 可以長講、自由度高，不用顧當下格式。
     - Whisper 在 GPU 上可以用 chunking 做到還不錯的即時性；就算不是即時，也可以幾分鐘內完成。
   - 缺點：
     - 是「批次」而不是「立刻回覆」，適合收集素材，不適合問答。

3. **未來：即時語音串流 → 本機 Whisper / 其他 ASR**
   - 若在 Windows + RTX 5080 上跑：
     - 可用的方案包含：
       - 直接跑 Whisper v3/v4 之類的模型（需自行管理 chunk & streaming）。
       - 使用專門的 streaming ASR 服務（例如一些商業 API 或 Groq 這類加速方案）。
     - 在有 GPU 的情況下，每句話做到 <300ms 延遲是可行的，重點在於：
       - 把音訊切成小片段、邊錄邊轉。
       - 接受偶爾需要修正的錯字，換取即時性。

### 文字 → 文字

- 手機上直接打關鍵字或 bullet 到某個 Obsidian note：
  - 這是最穩、最低風險的輸入形式。
  - 對 Pi 來說，清楚的 bullet / checklist + `@pi` 比一長串未整理文字更好處理。

## 3.2 處理模式（Pi 怎麼接）

### 即時模式

- 流程：
  1. 我用語音鍵盤或通話系統，把話轉成文字訊息給 Pi。
  2. Pi 盡量在幾秒內回應（短句、問題、架構）。
- 特性：
  - 適合「釐清」與「互動」：例如我說不清楚時，Pi 追問關鍵資訊。
  - Token 消耗是跟著時間走的：聊越久花越多。
- 控制策略：
  - 把即時模式限制成「任務導向通話」：例如 10 分鐘內只處理一個主題（起研究架構 / 決定下一步）。

### 批次模式

- 流程：
  1. 我在散步／開車時錄多段語音。
  2. 系統在背景把它們全部轉文字、放進 `Voice-Inbox`。
  3. 我回到電腦前，跟 Pi 說：「請幫我把今天所有 Voice-Inbox 整理到這篇研究底下。」
- 特性：
  - 適合「收集 + 整理」，不需要及時來回。
  - Token 消耗比較集中：一次處理完一整天的素材。

## 3.3 落地到 Obsidian（Pi → Obsidian）

1. **研究主檔（research note）**
   - 例如這個主題的主檔：統整所有子頁面的目錄與摘要。

2. **子頁面 / 子模組**
   - 每個主要面向（情境、技術構面、流程設計、可行性分析）拆成獨立檔案。
   - 好處：
     - 減少單檔長度，便於在手機上查看。
     - 不同 LLM / 不同任務可以對不同子檔案「上工」。

3. **標記規則（@pi / Pi TODO）**
   - 在任意段落插入：
     - `@pi 查資料：「關鍵字」` → 表示要我去網路上找資料回填。
     - `@pi 展開這一段` → 表示由 LLM 把該段落擴寫成完整小節。
   - 我收到指令後，可以：
     - 讀該段落上下文；
     - 查外部資料（若需要）；
     - 把結果直接寫回原檔案、或新增子小節。

## 3.4 最小可行版本（MVP） vs 進階版本

### MVP：
- 輸入：
  - 語音備忘錄 → JPR+iCloud → Whisper → `Voice-Inbox`。
  - 手機簡短語音輸入 → 直接傳給 Pi 的文字訊息。
- 處理：
  - 以「批次模式」為主，晚上統一請 Pi 整理。
- 落地：
  - Pi 只需要會讀 / 寫 Obsidian 的 `.md` 檔，依照你指定的研究檔案更新內容。

### 進階版本：
- 輸入：
  - 加上 Windows + 5080 上的即時 ASR，嘗試做「打電話給 Pi」式的體驗。
- 處理：
  - 加入即時對話模式，讓 Pi 在你散步 / 開車時能立刻追問細節。
- 落地：
  - 增加自動標記與自動分檔，例如：
    - 不同研究主題用不同前綴或 tag。
    - 自動把新素材歸檔到對應的 research 資料夾裡。
