# 04 研究問題與流程設計

> 本頁專注：列清楚研究問題，並草擬從 0 到完成研究的一條流程線。

## 4.1 研究問題整理（展開版）

1. **溝通方式問題**
   - 在散步／開車時，如果不能打字，我到底用什麼方式最穩定地把想法交給 Pi？
     - 選項：
       - A. 鍵盤語音輸入 → 文字訊息。
       - B. 長篇語音備忘錄 → 批次轉文字。
       - C. 類「打電話」的實時語音對談。
   - 「打電話式的即時對談」 vs 「短語音快照」：
     - 打電話：互動感強、可即時釐清，但對網路、token、系統整合要求高。
     - 短語音快照：實作簡單、對安全與專注比較友好，但需要回家後再整理。

2. **Obsidian 協作問題**
   - 當 Pi 已經幫我在 Obsidian 開好一篇研究，之後要怎麼設計「工作分工」？
     - 我負責：提出問題、補充例子、做最後判斷與改寫。
     - Pi 負責：整理素材、查資料、設計架構、寫草稿、產出摘要。
   - 我在 note 裡要怎麼標記關鍵字／TODO，讓 Pi 可以自動去查 Google、補寫內容？
     - 一種想像：
       - `@pi 查資料：「voice to obsidian workflow」`。
       - `@pi 展開：這段只是關鍵字，請寫成 3 段說明。`

3. **研究流程設計問題**
   - 從 0 到完成一篇研究，流程應該長什麼樣子？
   - 每一階段（出題 → 收集 → 架構 → 查資料 → 草稿 → 修稿）Pi 各能幫上什麼忙？

4. **成本與可維護性**
   - 什麼樣的設計不會太依賴單一平台（方便未來遷移）？
   - 怎麼設計才能在「好用」和「省 token / 降複雜度」之間取得平衡？

## 4.2 研究流程 v1（from 0 to 1）

### 第 0 階段：建立研究主檔
- 行為：
  - 我決定一個主題，請 Pi 在 Obsidian 的 `Research/` 底下開一篇主檔（像這個）。
- Pi 的工作：
  - 建立基本架構：Brief、情境、技術構面、研究問題、TODO。
  - 適度加入自己的理解與建議（不是只有空白模板）。
- 完成定義：
  - 有一篇可持續維護的主檔，之後所有素材都能收斂到這裡。

### 第 1 階段：出題（Problem Framing）
- 行為：
  - 我在散步／開車／桌前，用語音或文字講出：
    - 為什麼我要研究這個？
    - 我現在遇到的具體痛點？
- 工具：
  - 語音備忘錄、語音輸入、直接在主檔補充文字。
- Pi 的工作：
  - 把這些描述整理成 1–2 段清楚的「研究問題敘述」。
- 完成定義：
  - 研究問題能夠被一個外人理解，且我自己看了覺得「有說中我的痛」。

### 第 2 階段：收集（Capture）
- 行為：
  - 我在生活中想到相關的例子、想法，就用任何管道（語音、文字）記下來。
- 工具：
  - `Voice-Inbox/`、聊天紀錄、同一篇研究 note 底下的 My Notes 區塊。
- Pi 的工作：
  - 定期把這些零散素材整理成：
    - 短 bullet；
    - 關聯到研究問題的註解；
    - 或新的子研究問題。
- 完成定義：
  - 主檔中有一節「已收集素材」，我能一眼看到自己累積了哪些想法，而不是散落在各 App。

### 第 3 階段：架構（Structuring）
- 行為：
  - 我告訴 Pi：「差不多可以做第一次整理了。」
- Pi 的工作：
  - 根據現有素材：
    - 草擬一個 3–5 章節的研究大綱；
    - 每一章節底下列出預期要回答的問題。
- 完成定義：
  - 有一份我看得懂、覺得合理的大綱；
  - 每一章節都有簡短說明「這一節要幹嘛」。

### 第 4 階段：查資料（Research）
- 行為：
  - 我在各章節底下標記 `@pi 查資料` 任務，明確給關鍵字與需求（例如對比兩種方案、找案例）。
- Pi 的工作：
  - 對每個 `@pi` 任務：
    - 查資料（Google / 文件 / 實作經驗）；
    - 回填重點 bullet；
    - 若需要，附上 1–2 段解釋。
- 完成定義：
  - 大綱裡每一個重要節點，都有最少量的背景資訊支撐，不是空殼子。

### 第 5 階段：草稿（Drafting）
- 行為：
  - 我挑一個章節，請 Pi 根據現有 bullet / 資料，寫出第一版敘事文（可以是 blog 風格、技術說明、或設計文件）。
- Pi 的工作：
  - 寫字數適中的草稿；
  - 保留明顯段落與小標，使我容易修改。
- 完成定義：
  - 每個核心章節都有初稿，整篇研究串起來已經讀得通。

### 第 6 階段：修稿與總結（Revision & Synthesis）
- 行為：
  - 我閱讀全文，留言、改寫、補充自己的觀點；
  - 必要時再次請 Pi 協助改寫段落或重組結構。
- 工具：
  - Obsidian 的編輯 + 可能搭配 NotebookLM / Gemini 語音解說輔助理解。
- Pi 的工作：
  - 依照我的註解修稿；
  - 最後幫我產出摘要（1 段、1 頁、或投影片式重點）。
- 完成定義：
  - 我對這個研究有一個「可以再次使用」的成果：
    - 一篇文章、一份設計文件、或一組 SOP。
