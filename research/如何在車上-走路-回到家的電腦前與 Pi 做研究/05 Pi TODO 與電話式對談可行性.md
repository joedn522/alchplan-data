## 5.2 Windows + RTX 5080：能不能實現「打電話給 Pi」？

> 目標畫面：我用手機或電腦像打電話一樣跟 Pi 說話，Pi 能即時聽懂、思考、用語音回我，而且整個流程在成本與穩定度上是可接受的。

### A. 核心組件

1. **ASR（Automatic Speech Recognition）——語音轉文字**
   - 選項：
     - 本機 Whisper（medium / large v3 之類的模型）。
     - 專門的實時 ASR 服務（商業 API 或 Groq 加速的 Whisper）。
   - 對 5080 這種 GPU：
     - 完全有能力在 <300ms / 句 的延遲下運作，只要做 streaming chunk。

2. **LLM —— Pi 的「大腦」**
   - 選項：
     - 雲端 LLM（現在的我，以及你常用的 codex / sonnet / pro）。
     - 未來也可以考慮本機模型（但效果 vs 成本要權衡）。

3. **TTS（Text to Speech）——文字轉語音**
   - 選項：
     - 開源 TTS（例如 Coqui TTS、XTTS 等）。
     - 雲端 TTS API（商用服務）。
   - 5080 對 TTS 也相當有餘裕，可以做到接近即時的語音輸出。

4. **通話 UI / Phone Layer**
   - 實作方式：
     - A. 類 Discord/自製小程式：按下「開始對話」後，持續串流麥克風音訊給 ASR。
     - B. 真正的電話（SIP / VoIP）→ 伺服器 → ASR/LLM/TTS → 回撥語音。
   - 對你來說，MVP 大概會長成「耳機 + 桌面/手機 App」，不一定要真的走電話網路。

### B. 流程與延遲估算（概念級）

1. 我說話 → 本機 ASR 開始轉文字（0–300ms）。
2. 文字累積到一小句後送給 LLM（網路 RTT + LLM 推理：500–2000ms 視複雜度）。
3. LLM 回覆文字 → 本機 TTS 轉語音（100–500ms）。

- 這樣一來，從我說完一句話到聽到 Pi 回應，大約是 1–3 秒級別，對「聊天+思考」是可接受的。
- 真正的瓶頸在：
  - 網路穩定度；
  - LLM 回應的長度與模型大小（模型越大、答案越長，時間越久）。

### C. Token 與成本考量

- 假設一場 20 分鐘的「電話」：
  - 如果每 5–10 秒就有一個短問答，則整場對話可能會產生數千到上萬 token；
  - 相比之下，一封整理好的「語音快照」+ 一次長回覆，token 成本會小很多。
- 這也是為什麼：
  - 我會建議把「打電話給 Pi」定位成：
    - 用在關鍵的釐清／設計會議；
    - 不用每天、也不需要長時間掛著。

## 5.3 Pi 文字 → Gemini 語音 Channel 的模式

### A. 概念設計

1. Pi 角色：
   - 背景研究員：先幫你整理需要釐清的問題、關鍵重點、可能答案。
2. Gemini 語音 Channel：
   - 對話前端：你和 Gemini 透過語音聊，針對那些重點做「互動式理解」。
3. 回寫 Obsidian：
   - 對談後，請 Gemini 產出 summary / Q&A 紀錄，貼回研究 note。

### B. 範例流程

1. 你說：「Pi，幫我整理這篇研究裡我還搞不懂的 5 個問題，寫成給 Gemini 的議程。」
2. Pi 在研究 note 裡生成一段：
   - 「Gemini 對話議程：問題 1、2、3…，請逐一跟我確認。」
3. 你把這段貼進 Gemini 的語音 Session 開場白，開始語音聊天。
4. 對話完後，請 Gemini 幫你寫一份「這次對談的摘要與決策」，貼回 Obsidian。
5. Pi 再讀這份摘要，幫你更新研究檔中的結論區塊。

### C. 優點 / 缺點

- 優點：
  - 利用現成的語音對話能力，不用自己搭整套通話基礎建設。
  - Pi 專注於整理、設計議程、收斂成果。
- 缺點：
  - 流程多一步（Pi → Gemini → 你 → 回 Pi）。
  - 需要有意識地把成果「帶回 Obsidian」，不然會散在不同產品裡。

## 5.4 `@pi` / `Pi TODO` 標記規則（初稿）

> 目標：讓未來的我在任何研究 note 裡，只要打一行簡單指令，就知道是要交給 Pi 哪一類任務。

### 任務類型範例

1. **查資料**
   - 語法：
     - `@pi 查資料：「關鍵字或問題描述」`
   - 行為：
     - Pi 查 2–5 個可靠來源，整理成 bullet + 簡短說明。

2. **展開大綱**
   - 語法：
     - `@pi 展開：` + 一串關鍵字或粗略段落。
   - 行為：
     - Pi 將其轉成 2–4 段連貫的文字，保留小標。

3. **補完段落 / 改寫**
   - 語法：
     - `@pi 改寫：` + 某段文字；
     - 或直接在段落旁標註「@pi 請幫我變得更口語／更技術／更精簡」。

4. **研究任務管理**
   - 語法：
     - `Pi TODO:` 開頭的 checklist；
   - 行為：
     - Pi 定期掃這些 TODO，完成後在旁邊附上「完成日期 + 簡短說明」。
